## 2. 差分の要点

### 2.1 クリティカルパス・初回応答

- **FREE AGENDA**: クリティカルパス **599 ms**、初回 HTML 7.20 KiB。Next.js のサーバー/Edge 処理と複数リソースの依存関係の分、チェーンが長い。
- **tokura.app**: クリティカルパス **46 ms**、初回 **2.02 KiB** のみ。ほぼ静的 HTML 1 本で描画が完結しており、待ち時間が極端に短い。

→ **速度差の主因の一つは「初回ドキュメントの軽さとクリティカルパスの短さ」**（tokura.app はミニマルな静的ページ）。

### 2.2 転送サイズと「推定削減」

- FREE AGENDA: 38.6 KiB で、リソース数・継続時間に対しては妥当な水準。
- tokura.app: 78.1 KiB だが、**推定削減約 59 KiB** の指摘あり（画像最適化などで削れる余地があると判断されている）。実効的には軽い構成で 100 点を達成。

### 2.3 LCP 要素

- **FREE AGENDA**: LCP 候補は `body > main > div > … > div.content-below-fold` 付近（CLS 0.016 の要素）。ヒーロー画像は `next/image` で最適化済みだが、レンダリング順やフォント/JS の影響で LCP が画像以外になっている可能性あり。
- **tokura.app**: LCP 要素は `main > section > div > img`（`logo-large.png`）。ロゴ画像 1 枚で LCP が決まるシンプルな構成。

### 2.4 パフォーマンススコア

- モバイルで **91 vs 100** の差。FREE AGENDA は 91 で「良好」だが、クリティカルパス 599 ms と初回チェーンの重さがスコアを下げている。
- デスクトップでは FREE AGENDA も 100。ネットワーク・CPU に余裕がある環境では問題になりにくい。

---

## 3. 速度差の要因まとめ（3〜5 個）

1. **初回 HTML サイズとクリティカルパスの長さ**  
   tokura.app は 2.02 KiB の HTML と 46 ms のクリティカルパス。FREE AGENDA は 7.2 KiB と 599 ms で、Next.js・Edge・複数リソースの依存が効いている。

2. **ページ構成のミニマルさ**  
   tokura.app はロゴ＋検索 UI 中心の静的寄り構成。FREE AGENDA は検索・フォント・GA・トップローダーなどリソースが多く、パース・実行・描画のコストが増える。

3. **LCP までのリソース数**  
   tokura.app は LCP がロゴ画像 1 本にほぼ限定。FREE AGENDA はフォント・JS・画像など複数が LCP 前にかかり、クリティカルパスが伸びやすい。

4. **サーバー/Edge の応答とチェーン**  
   FREE AGENDA は「ドキュメント → 続くリソース」のチェーンが長く、合計 599 ms。tokura.app は初回 1 本で 46 ms に収まっている。

5. **転送量より「何をいつ読むか」**  
   tokura.app は転送 78.1 KiB でも、初回描画に必要なものが少なく 100 点。FREE AGENDA は 38.6 KiB と少なめだが、初回に必要なリソースの数と順序がスコアに効いている。

---

## 4. FREE AGENDA 側の改善の方向性（今回の結果を踏まえて）

- **クリティカルパス短縮**: 初回 HTML をできるだけ小さくし、必須 CSS/JS をインライン化または削減。フォントの `display: swap` や preload の見直し。
- **LCP の安定化**: ヒーロー画像を S3/CloudFront 化するか、現状の `next/image` のまま `priority`・`quality`・`sizes` を最適化し、LCP が画像になるよう調整。CLS 0.016 は許容範囲だが、レイアウトシフトの原因を減らす。
- **計測の継続**: 変更のたびに PageSpeed Insights（モバイル）で「クリティカルパスの最大待ち時間」「初回ドキュメントのサイズ・時間」を記録し、91 → 95+ を狙う。
